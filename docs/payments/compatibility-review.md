## 1. "990원 결제 후 여러 번 보임" — 버그인가 설계인가 → **설계 (코드 정상)**

### 근거
- 궁합 결과는 **`love-question`** 권한으로 게이팅됨:
  - `src/app/compatibility/result/page.tsx:122-124` · `input/page.tsx:43` → `getTasteProductEntitlement(user.id, 'love-question')` — **scopeKey 없이** 조회.
- `love-question` 의 scope = **global** (`src/lib/payments/product-scope.ts`): `love-question`/`money-pattern`/`work-flow` 는 `kind: 'global'`, `scopeKey: null`.
- 즉 **990원 1회 결제 → 글로벌 영구 권한 → 어떤 커플이든 무제한·영구 열람**. 코드는 설계대로 동작하는 것이며 **엔타이틀먼트 조회 오류가 아님**.

### 추가 발견 (중요)
- 궁합 전용 상품이 **따로 없음**. "연애 마음 확인"(`taste_love_question`)과 궁합이 **동일 상품을 공유**. → 990원으로 "연애 마음 확인"을 사면 **궁합도 같이 열리고**, 그 반대도 성립.
- `compatibility-result-view.tsx:438` 의 재결제 CTA(`product=love-question`)도 글로벌 권한이라 한 번 사면 다시 안 뜸.

### 결론
- "버그"가 아니라 **"990원 = 글로벌 영구 무제한"이라는 원래 설계**. 사용자가 기대한 "990원 = 1회"와 **설계가 불일치**.
- 1회 과금 모델로 바꾸려면 **scope 모델 변경**이 필요(아래 §3-A).

## 2. 내용이 부실한 이유 → **결정론 템플릿(LLM 미사용)**

- 궁합 해석(`src/lib/compatibility.ts`, 972줄)은 **오행 조합별 고정 템플릿**으로 생성:
  - `COMMUNICATION_STYLES` / `MONEY_STYLES` / `DISTANCE_STYLES` 등 오행(목·화·토·금·수)별 `summary`/`caution` 맵 + score + practical 카드.
  - 두 사람 오행 조합을 맵에서 꺼내 조립 → **입력이 같으면 항상 같은 짧은 문구**.
- 사주 평생풀이(`saju-lifetime-interpretation`)는 **OpenAI LLM 챕터**로 깊게 생성하는데, **궁합엔 LLM 경로가 없음**. 이 격차가 "부실"의 핵심.

### 개선안 (택1 또는 병행)
- **(빠름)** 템플릿 확장: 조합 수·문장 길이·관계유형(연애/친구/가족/사업)별 분기 보강. 비용 0, 깊이 제한적.
- **(권장)** **궁합 LLM 해석 도입**: 사주 챕터 모델 재사용 — 두 사람 사주 데이터 + 관계유형을 프롬프트로 깊은 해석 생성, 결정론 fallback 유지. 사주와 품질 균형. 단가/지연/캐시(분포당 1회) 설계 필요.
- 어느 쪽이든 **유료(990+) 화면이므로 결제 가치에 맞는 분량/구체성**이 핵심.

## 3. 가격/기능 제안 검토 — 990 1회 / 월 4,900 / 년 9,900 + 만료 알람

### 3-A. 모델 변경에 필요한 것 (설계 작업)
사용자 의도 = **990원=1회, 월/년=무제한**. 현 글로벌·영구 모델에서 다음이 필요:
1. **궁합 1회권 scope 신설**: love-question 글로벌 → 궁합은 **커플 단위 scope**(예: `compat:{readingA}:{readingB}`)로 분리. 990원=해당 커플 1건. (today-detail·lifetime 의 per-reading scope 패턴 재사용)
   - 연애 마음 확인(love-question)과 궁합을 **상품 분리** 필요(현재 동일 상품).
2. **월/년 무제한 = 시간 만료 권한**: 현재 `product_entitlements` 에 **`expires_at` 없음**(영구). 신규로:
   - `expires_at` 컬럼 + 조회 시 만료 체크 + 만료 후 재게이팅.
   - 또는 구독(`subscriptions`, Toss 빌링키 자동결제) 활용 — 단 자동갱신이라 "만료 알람" 취지와 다름.
3. **만료 알람**: VAPID 푸시 + cron(D-3/D-1/당일) 인프라는 **이미 존재**. 만료일 기반 알림 잡 추가로 구현 가능. 리텐션 장치로 효과적.

### 3-B. 제안 가격에 대한 판단 ⚠️ (가격 사다리 충돌 + 월/년 역전)
현 카탈로그와 충돌·역전이 있어 **그대로는 비추천**:

| 문제 | 내용 |
|---|---|
| **월/년 가격 역전** | 월 4,900 × 12 = 58,800 vs 년 9,900 → **년이 월의 83% 할인**. 두 달 이상 볼 사람은 무조건 년 구매 → **월 상품이 사실상 죽음**(아무도 월 안 삼). 정상 연할인은 ~2개월 무료(약 17%) 수준. |
| **9,900 중복** | 9,900 = 기존 **36코인팩 / 프리미엄 멤버십(월)** 과 동일가 → 사용자 혼동("9,900이 뭐였지"). |
| **4,900 중복** | 4,900 = **라이트 멤버십(월)** 과 동일가 → "궁합 월 4,900" vs "멤버십 월 4,900" 혼동. |

### 3-C. 권장 가격(가설 — A/B·운영자 확정 전)
- **1회(커플당)**: **990원** (현행, 단 진짜 1회로 scope 분리)
- **월 무제한**: **3,900원** (라이트 멤버십 4,900 아래로 차별화)
- **년 무제한**: **24,900~29,900원** (월의 약 6~7배 = 합리적 연할인, 9,900/49,000 과 겹치지 않음)
- 대안: 년을 꼭 9,900 으로 두려면 **월을 1,900원**으로 낮춰 월/년 공존(9,900 ÷ 12 ≈ 825 < 1,900, 여전히 연할인이 큼 — 이 경우 월은 "찍먹"용).
- 핵심 원칙: **월·년 가격차가 "연할인"으로 납득되고, 기존 9,900/4,900 상품과 안 겹치게**.

### 3-D. 리스크
- **기존 990원 글로벌 구매자**: 이미 영구 무제한 보유 → 1회 모델로 바꾸면 **grandfather 처리**(기존 권한 유지) 필요. 안 그러면 "샀는데 막혔다" 클레임.
- **결제 정합성**: 시간 만료 권한 + 구독 + 1회권이 섞이면 조회 우선순위 재설계 필요(P0 환불 회수도 만료권에 맞춰 확장).
- **콘텐츠 가치**: 무제한 상품을 팔려면 §2(내용 깊이)가 선행돼야 "무제한 볼 가치"가 생김. **내용 개선 없이 무제한만 추가하면 환불·불만 위험**.
