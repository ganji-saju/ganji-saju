# 나이스페이 "카드결제 전용" 제약 대응 — 코인/멤버십 재구조화 분석

> 작성: 2026-06-30 (Claude) · 근거: 코드 직접 조사(코인/멤버십/결제수단 맵) + 한국 결제규제 리서치
> ⚠️ 규제 항목은 **비단정·검증필요** — 법무/나이스페이 심사팀 확인 전 실행 금지. 코드 사실은 파일:라인 근거.

## 0. 한 줄 결론

진짜 막히는 건 **코인충전(선불충전)** 하나뿐. **멤버십·단건상품(점수/궁합/평생리포트)은 이미 "카드 1회 직접결제"라 카드전용 정책과 정합 → 그대로 생존.** 코인을 끄고, 코인으로 쓰던 3개 기능(상세풀이·달력·대화상담)을 이미 라이브인 카드 직접결제/패스로 옮기면 된다. 권고: **A안(코인 폐지+카드 직접결제) 베이스 + C안 일부(대화상담 턴팩/패스·멤버십 혜택 재표현) 결합.**

## 1. 현재 구조 (코드 확인 결과)

### 1-1. 멤버십 = 정기결제 아님, "30일권 1회 카드결제"
- `membership_plus` 4,900원/월, `membership_premium` 49,000원/월 — 둘 다 **빌링키 자동결제 아님**.
- 근거(추측 아님): `requestBillingAuth`/`v1/billing` 호출 전역 0건, `/api/billing` 빈 디렉토리, 자동 재청구 크론 없음, `expireIfNeeded`는 30일 경과 시 `status='expired'`로 만료만 시킴 → **사용자가 직접 재구매**. `subscriptions.toss_billing_key` 컬럼은 있으나 항상 null인 dead 스캐폴드.
- **함의: 멤버십 결제 메커니즘은 카드 1회결제라 카드전용과 충돌 없음. 손 안 대도 됨.** 단 혜택이 "코인 지급"(plus 2·premium 90)이라 코인 폐지 시 혜택만 재표현 필요.

### 1-2. 코인 = 선불충전(stored-value) — 문제의 핵심
- `credit_15`(9,900원/15코인), `credit_40`(19,800원/40), `credit_100`(44,900원/100) 카드 충전 → `credit_lots`(1년 만료)/`subscription_balance`(무만료) 잔액에서 `deduct_credits`로 차감.
- **코인을 실제로 차감하는 기능은 단 3개**: 상세풀이(10), 월간달력(10), 대화상담(3턴 묶음당 3코인). `compat`(10)·`daewoon`(3)은 **정의만 있고 실제 차감 미연결**.

### 1-3. 카드 직접 단건결제 인프라 — **이미 프로덕션 라이브**
- `taste_*`(상세풀이·연애·금전·직업·달력·연운·점수·궁합깊은풀이 등), `lifetime_report`(보관형 49,000원), `bundle`(오늘 풀세트 19,800원)는 전부 `credits:0`. 코인 안 거치고 `fulfillment.ts`에서 **엔타이틀먼트를 직접 부여**.
- 즉 **상세풀이·달력은 동일 기능의 9,900원 카드 단건상품이 이미 카탈로그에 존재** → 코인 경로만 떼면 그대로 흡수.
- 궁합(love-question/compat-reading)은 이미 카드 직접결제, 코인 경로 없음.

> ⚠️ 가격 표기 확인필요: 코드(catalog.ts)는 단건 9,900원으로 보이나, 과거 메모(점수 550원·궁합 990원)와 불일치. 프로덕션 실가격은 별도 확인.

## 3. 규제: "법적으로 막힘" vs "PG 정책상 막힘" (⚠️ 검증필요)

| 항목 | 성격 | PG 바꾸면? | 비고 |
|---|---|---|---|
| **코인/포인트 충전** | 혼합(법적+PG) | 부분적 | 선불전자지급수단. 자가형/소액예외(발행잔액 30억 미만 **AND** 연 발행 500억 미만)면 금융위 등록 면제. **초과 시 등록의무(미등록=5년 이하 징역/3억 이하 벌금)**. 2024.9 개정 전금법 강화 |
| 정기결제/빌링 | PG정책 | 부분적 | 법적 금지 근거 없음. 무형콘텐츠+충전결합이라 심사난도↑. 조건 충족 시 일부 PG 가능 |
| 계좌이체·가상계좌 | PG정책 | 부분적 | 비가역·선입금 환불리스크 관리. 법적 금지 근거 미확인 |
| 휴대폰결제 | 혼합 | 부분적 | 통신과금(전기통신사업법) 별도. 운세는 정보이용료로 흔함 → 다날 등 특화사 여지 |
| 사주업종 "카드전용" 일반사유 | PG정책 | 예 | 무형콘텐츠 환불·민원·환금성 리스크 → PG가 카드로 좁힘. **법 제약 아님** |

**요지:** 카드전용 강제의 1차 원인은 **PG 업종심사/리스크 정책**(다른 PG면 일부 수단 열릴 여지). 단, **코인충전만은 진짜 법적 층(선불전자지급수단)** 이 얹혀 있어 PG를 바꿔도 구조적 위험이 남는다. → 그래서 코인은 "다른 PG로 우회"보다 "모델 전환"이 정석.

## 4. 옵션 비교

| | A. 코인 폐지+카드 직접결제+기간권 | B. 코인 유지+PG/구조 변경 | C. 묶음·기간권 중심 재설계 |
|---|---|---|---|
| 개념 | 코인 폐지, 3개 기능을 카드 단건/패스로 | 충전 허용 PG(다날·갤럭시아·KCP 등)로 이전 | 코인 대신 "기능·기간 한정 환금불가 패스"(올액세스·턴팩) |
| 개발공수 | **중(2~4주)** — 직접결제 인프라 이미 존재 | 소~중(코드는 작으나 PG 심사 리드타임 수주~수개월) | 중~대(3~6주, 패스 차감·부분환불 신설) |
| 법적리스크 | **낮음**(stored-value 제거=근본해소) | 중~높음(선불수단 성격 잔존, 규모↑시 형사) | 낮음~중(범용화되면 다시 회색지대) |
| 매출영향 | 중립~소폭하락(충전 락인 상실) | 긍정(BM 연속성) 단 승인 불확실 | 긍정 잠재(LTV·락인↑) |
| 핵심 단점 | 대화상담 micro-UX, 락인 상실 | 근본원인 미해소→재차단·형사 가능 | 설계·회계 복잡 |

## 5. 권고 (단정 아님)

**A안 베이스 + C안 핵심 2개(대화상담 턴팩/패스 · 멤버십 혜택을 정액 쿼터로 재표현) 하이브리드.**

근거:
1. 진짜 법적 층은 **코인 한 곳**뿐 + 카드 직접결제·엔타이틀먼트 인프라가 **이미 라이브** → 폐지·전환 비용 구조적으로 낮음. 상세풀이·달력은 동일 9,900원 단건상품이 이미 있어 코인 경로만 제거.
2. **B는 임시방편** — 나이스가 막은 근본원인(환금성+선불수단 법적층)을 안고 가므로 다른 PG도 동일 사유로 재차단·반려 가능, 규모 성장 시 등록의무(형사) 발생.
3. 순수 A의 약점(대화상담 소액반복 UX + 충전 락인 매출)은 **C의 턴팩/기간권**으로 정확히 보완. 멤버십은 결제 메커니즘 그대로 생존, 혜택만 재표현.

### 실행 순서(제안)
- **단기(저위험·즉시):** 코인 **신규 충전 중단**(가장 위험한 한 곳 차단) + credits 페이지/카피 정리.
- **중기:** 코인 차감 3기능을 직접결제/패스로 전환 → 멤버십 혜택을 정액 쿼터로 재표현 → 기존 보유 코인잔액 소진 유예 또는 환불 마이그레이션.
- **그 후:** 정기결제·비카드 수단(휴대폰결제 등)은 **별도 PG 트랙**으로 분리 검토.

## 6. 실행 전 반드시 확인 (법무/PG)

1. **나이스페이 심사팀**: 비카드/충전/정기결제 차단이 "업종심사 정책"인지 "선불수단 법적 등록요건"인지 사유코드 직접 질의 (계약서/심사회신으로만 확정).
2. **전자금융 전문 변호사**: 현 코인 BM이 선불전자지급수단 등록의무 대상인지, 자가형/소액예외 범위인지 (미등록 영업=형사처벌).
3. **C 패스 설계 경계**: 어디까지가 "이용권"이고 어디부터 "선불수단"인지(단일기능 한정 vs 다기능 범용·환금성·유효기간) — 법무 사전검토.
4. **기존 코인잔액 처리**: 환불 vs 소진유예 vs 전환의 전자상거래법 환불의무·회계·CS.
5. **타 PG 승인조건**(다날·갤럭시아·키움페이·토스·KCP): 코인충전/빌링/계좌이체/휴대폰결제 각 업종 승인 가능성·조건, 게재 직전 재확인.
6. **소액 카드결제 채산성**: 대화상담 단가 vs PG 고정수수료 — micro-transaction 마진 확인.
7. 가격 표기 불일치(코드 9,900 vs 메모 550/990) 및 `credits/page.tsx` 카피("연애 마음 확인 10코인" — 실제 코인 차감 없음) 정합성.
