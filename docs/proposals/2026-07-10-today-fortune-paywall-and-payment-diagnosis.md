## 0. 한 줄 결론

**"무료가 너무 좋아서 결제를 안 한다"는 가설은 데이터로 지지되지 않는다.** 결제까지 도달한 사람이 사실상 없고, 도달한 사람은 **서버 오류로 실패**했다. 결제 성공은 서비스 역사상 **0건**이다. 페이월을 조이기 전에 결제를 고쳐야 한다.

## 2. 무료 vs 유료, 지금 실제 경계 (코드 근거)

### 무료가 주는 것
총운 점수 + 등급, 6축 점수 그리드, 오늘 해볼 일/줄일 일 각 1개, "왜 이렇게 보이나요" 요약(factLines 3개), **일진 8영역 점수 산출내역 + 합산식 전체**(`today-iljin-breakdown-card.tsx`), 카테고리별 풀이, **행운 패키지 12종**, 명식 4기둥 차트.

### 유료(`today-detail`, 9,900원)가 **추가로** 주는 것
- 12시진 **시간대별 길흉 가이드** (`favorableWindows` / `cautionWindows`)
- 행동 리스트 확장 (`recommendedActions` / `avoidActions`)
- 시나리오 비교 (`scenarios`)
- **풀이 근거 전체** (`evidenceLines` — 무료는 3줄 요약만)
- 일진 메시지 3개 → 5개
- **AI 깊은 풀이** (`aiNarrative`, LLM 1회 생성·영속)

### 여기서 진짜 문제
유료 증분이 **무료 화면에서 아예 보이지 않는다.** `PremiumLockCard` 는 콘텐츠를 렌더하지 않고 CTA 카드만 띄운다. 사용자는 **자기가 무엇을 사는지 모른 채** 9,900원을 결제해야 한다.

무료가 과한 게 아니라, **유료가 안 보이는 것**이다.

## 3. 파급효과 분석 (무료 축소 + 블러 + 결제 팝업을 지금 실행할 경우)

### 1차 효과 (누구나 예상)
- 무료 결과 화면의 정보량 감소, 결제 레이어 노출 증가
- 결제 시도(클릭) 건수 증가
- 무료 이용자의 즉각적 불만

### 2차 효과 (1차의 결과)
- **습관 파괴**: 오늘운세는 매일 오는 습관 상품이다. 무료 결과가 만족스러워서 재방문하던 이용자의 방문 이유가 사라진다. 84건의 `readings` 는 18명이 만든 것이고, 재방문이 유일한 자산이다.
- **유일한 검증된 강점 소각**: 사용자가 자발적으로 준 피드백이 "무료가 너무 잘 나온다"이다. 이건 제품이 통했다는 **유일한 긍정 신호**다. 경쟁 앱이 널린 시장에서 그걸 없애면 차별화가 사라진다.
- **공유/유입 감소**: 공유 스냅샷(궁합·오늘운세)이 유입 경로인데, 공유받은 사람이 보는 화면이 블러투성이면 공유 자체가 줄어든다.

### 3차 효과 (예측 불가 영역)
- 가입 0인 상태에서 결제 레이어만 늘리면, **비로그인 이용자가 결제 벽에서 이탈**한다. 지금 가입 전환이 0이라는 건 로그인 요구 자체가 이미 벽이라는 뜻이다. 벽을 하나 더 세우는 셈.
- 트래픽이 하루 150~780건이고 최근 7일 결제 시도가 1건이다. **A/B 테스트로 페이월 효과를 측정할 표본이 없다.** 바꾼 뒤 좋아졌는지 나빠졌는지 알 수 없고, 결국 "느낌"으로 더 조이게 된다.

### 의도치 않은 결과 (가장 유력)
결제가 깨져 있는 상태에서 블러를 늘리면, **결제 의향이 있는 소수를 모아 전부 `"사용자 정보가 존재하지 않습니다"` 오류로 떨군다.** 돈을 내려다 실패한 사용자는 그냥 안 산 사용자보다 훨씬 강하게 이탈하고, 다시 오지 않는다. 가장 귀한 고객군을 태우게 된다.

### 피드백 루프 (가장 중요)
**무료 축소 → 재방문·공유 감소 → 유입 감소 → 결제 모수 감소 → 전환율이 더 나빠 보임 → "더 조여야겠다" → 무료 추가 축소.**
이 루프는 스스로를 강화하며, 각 단계마다 "역시 무료가 문제였다"는 잘못된 확증을 준다.

## 4. 권고 — 순서가 전부다

### P0. 결제 확정 실패 규명 (이거 없이는 나머지가 무의미)
`payment_failed` 4건의 `last_error = "사용자 정보가 존재하지 않습니다."` 를 재현·수정한다. 가설(⚠️ 검증필요): PG 리다이렉트 복귀 시 세션/쿠키가 유실돼 서버가 사용자를 못 찾는다. `payment_funnel_events` 에 `confirm_*` 이 0건인 것이 이 가설과 정합적이다.

**성공 기준**: 실결제 1건이 `status='paid'` 로 기록되고 `product_entitlements` 가 부여된다.

관련 기존 이슈: 나이스페이 계좌이체 W004(키 미연결), clientKey 끝 탭 문자(U116). `TRANSFER` 주문 4건이 전부 `prepared` 에 멈춰 있는 것과 연관 가능.

### P1. 가입 병목 (7일간 방문 2,540건, 가입 0건)
결제에 로그인이 필요한데 가입 전환이 0이다. 페이월보다 이 벽이 먼저다.

### P2. 측정 가능하게 만들기
현재 클라이언트 퍼널(`premium_teaser_viewed` → `unlock_clicked` → `premium_result_viewed`)과 서버 퍼널(`payment_funnel_events`)이 **조인되지 않는다.** 무료 결과 조회 대비 결제 전환율을 지금은 계산할 수 없다. 페이월을 바꾸기 전에 이걸 먼저 이어야 변경의 효과를 알 수 있다.

### P3. 그 다음에 페이월 재구성 — 방향을 뒤집어서

| 항목 | 제안 | 이유 |
|---|---|---|
| 무료 콘텐츠 | **깎지 않는다** | 검증된 유일한 강점. 습관·공유의 원천 |
| 블러 | **도입한다. 단 대상은 "지금 아예 안 보이는 유료 증분"** | 블러는 감추는 장치가 아니라 **살 물건을 보여주는 장치**다 |
| 블러 대상 | 시간대별 길흉 가이드, 행동 리스트, 시나리오, 근거 전체, AI 서사 | 무료엔 원래 없던 것들 — 박탈감 0, 구매 동기 ↑ |
| 결제 레이어 | **모달 팝업 ✗ / 인라인 블러 + 하단 고정 CTA ✓** | 시니어 타깃(#625). 모달은 뒤로가기·스크롤·접근성에서 취약. 기존 body portal dock 패턴 재사용 |
| 재사용 | `ScoreLockGate`(`src/components/saju-score/score-lock-gate.tsx`) | 이미 블러+오버레이+가격칩+CTA 구현됨. CTA href/문구가 score-total 하드코딩이라 파라미터화 필요 |

`ScoreLockGate` 는 잠금 시 **children 을 아예 렌더하지 않는다**(민감값이 DOM/번들에 남지 않게 — PR #311 교훈). 유료 본문 블러에 재사용할 때 이 성질을 유지해야 한다. 즉 "진짜 내용을 흐리게" 가 아니라 "형태만 흉내낸 스켈레톤을 흐리게" 다.

## 5. 점수 산출내역을 보여줄 필요가 있는가

**현재**: 일진 8영역(천간·지지·오행·신살·균형·조후·운성·특수) 점수와 **합산식까지 무료로 전면 공개**.

**의견**: 산출내역은 신뢰를 만드는 장치이지 가치를 만드는 장치가 아니다. 다만 **숫자 합산식 공개는 신뢰가 아니라 반증거리를 만든다.**

- "왜 신살이 3점인가?" 에 답할 수 없다. 답하려면 결정론 템플릿의 계수를 설명해야 한다.
- 숫자 합산은 운세를 **계산기처럼** 보이게 한다. 신비감과 서사가 이 상품의 값어치인데, 그걸 스스로 깎는다.
- 반대로 **서술형 근거**("일간이 약한데 오늘 재성이 강해 지출 유혹이 있다")는 신뢰를 만든다.

**권고**:
1. 8영역 숫자 breakdown 은 **기본 접힘(details)** 으로 내린다. 원하는 사람만 편다. 삭제는 하지 않는다.
2. 서술형 근거(`factLines`)는 무료에 **그대로 유지**한다.
3. 근거 전체(`evidenceLines`)는 **지금처럼 유료**로 둔다. 이미 적절한 경계다.
4. 산출내역을 **유료로 잠그는 것은 반대**한다. 근거를 감추면 신뢰가 깎이고, 신뢰는 결제의 전제다. 감출 것은 근거가 아니라 **깊이**다.

## 6. 요약

- 페이월을 조이는 결정은 **지금 내리면 안 된다.** 결제가 0건 성공이고, 원인은 서버 오류로 보이며, 효과를 측정할 표본도 없다.
- 사용자가 "무료가 너무 좋다"고 말하는 것은 **문제가 아니라 자산**이다. 문제는 그 다음에 살 것이 보이지 않고, 사려 해도 결제가 실패한다는 것이다.
- 블러는 도입하되 **무료를 덮는 게 아니라 유료를 드러내는 방향**으로.
- 순서: 결제 수리 → 가입 병목 → 측정 연결 → 페이월 재구성.
