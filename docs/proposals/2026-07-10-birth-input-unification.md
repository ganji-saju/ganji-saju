# 출생정보 입력 개선 제안 — 오늘운세·사주 통합 + 사주 위저드 단순화

> 작성 2026-07-10 · 상태: 제안(미승인) · 트리거: 시니어 사용성 피드백
> "오늘운세·사주 입력창이 각각이라 불편 / 사주 3번 넘기기 불편 / 입력창이 너무 많다"

## 1. 현황 진단 (코드 근거)

### 1-1. 두 폼이 별개 상태·별개 UX
| 항목 | 오늘운세 | 사주 |
|---|---|---|
| 진입 | `/today-fortune` | `/saju/new` |
| 컴포넌트 | `BirthInfoStepper` + `CompactBirthFields` (단일 패널) | `saju-intake-page.tsx` (1713줄, 3스텝 스와이프 위저드) |
| 상태 타입 | `TodayFortuneBirthPayload` (14필드, 이름 없음) | `SajuOnboardingDraft` (별칭 포함) |
| 화면 수 | **1패널 스크롤** | **3스텝 넘김** (birth→location→profile) |
| 게스트 저장키 | (오늘운세 전용) | `moonlight:saju-recent-guest-input-v1` |
| 로그인 프로필 | `/api/profile` 자동로드·저장 | `/api/profile` 자동로드·저장 |

**핵심**: 밑단 리졸버(`resolveUnifiedBirthInput`), 프리셋(`BIRTH_LOCATION_PRESETS`), slug(`toSlug`)는 **이미 공통**이다. 갈라진 건 (a) 상태 타입 2종, (b) **게스트 저장키가 서로 달라 재입력 발생**, (c) 사주만 3스텝.

### 1-2. 사주 "입력창 너무 많다"의 실체
- 필수 입력은 사실상 **생년월일·시각·성별·출생지** 4종뿐인데, 스텝1(별칭+생년월일+양력음력+시각+성별)과 스텝2(출생지+시간룰)로 **쪼개져** 있어 많아 보인다.
- 스텝3(관심주제·관계·직업·고민)은 **전부 선택사항**인데 별도 스텝으로 존재해 "한 번 더 넘김"을 강제한다.
- 스텝 전환 주동작이 **좌우 스와이프**(54px 임계) — 시니어가 발견/조작하기 어렵다. CTA 버튼은 있으나 보조.

## 2. 개선 방향 (3옵션)

| 옵션 | 내용 | 화면 넘김 | 재입력 | 비용 |
|---|---|---|---|---|
| **1 (최소)** | 게스트 저장키 통일(공용 `birth-profile` 1개) → 오늘운세·사주 상호 자동채움 | 사주 3 유지 | **제거** | 낮음 |
| **2 (추천)** | 옵션1 + 사주 스텝1·2 병합(1화면) + 스텝3(관심/상황)을 결과 이후로 이동 + 시니어 UI(큰 버튼·프리셋 우선·스와이프는 보조) | 사주 **0~1** | **제거** | 중간 |
| **3 (야심)** | 옵션2 + 단일 입력 허브: 1회 입력으로 오늘운세·사주 진입 선택 | 통합 | 제거 | 높음(IA 변경) |

**추천: 옵션 2.** 세 가지 불만(각각 입력/3번 넘김/입력 과다)을 모두 직접 해소하고, 공통 모듈이 이미 있어 위험이 낮다. 옵션 3은 후속.

## 3. 옵션 2 상세 설계

### A. 사주 위저드: 3스텝 → 1필수 화면
`saju-intake-page.tsx`의 `BASE_STEPS`(birth, location) 두 스텝을 **하나의 "기본 정보" 화면**으로 병합. `UnifiedBirthInfoFields`를 전 섹션(`date`+`gender`+`location-time`) 한 번에 렌더 — 오늘운세가 이미 쓰는 단일 패널 패턴과 동일.

한 화면 구성(위→아래, 큰 라벨·큰 입력):
1. 이름(별칭)
2. 생년월일 년/월/일 + 양력/음력 토글
3. 태어난 시각 + "시간 모름"
4. 성별
5. 출생지 — **프리셋 칩(모름·서울·부산…) 먼저**, 검색은 접기(펼침)
6. 하단 큰 버튼 **"사주 풀이 보기"**

- 스텝3(관심주제/관계/직업/고민)은 전부 선택사항 → **필수 흐름에서 제거**. 결과 화면에서 "더 정확하게 보기" 접이식 옵션으로 이동(원하면 보강, 안 해도 결과 나옴).
- 결과 → "3번 넘김"이 **0번**(스크롤 1회)으로.

### B. 게스트 정보 공유: 공용 저장키 1개
- 신규 공용 키 `moonlight:birth-profile:last`(가칭)에 마지막 출생정보 저장.
- 오늘운세·사주 **양쪽이 이 키를 읽고 쓴다** → 한 곳에서 입력하면 다른 곳 자동 채움.
- 로그인 사용자는 기존 `/api/profile` 경로 유지(이미 공유됨). 게스트만 이 키로 보강.
- 2회차 진입 UX: "○○님 정보로 볼게요 · [정보 바꾸기]" — 재입력 없이 바로 진입.
- 마이그레이션: 기존 `saju-recent-guest-input-v1` 값이 있으면 최초 1회 새 키로 흡수.

### C. 시니어 배려 (양 폼 공통)
- 스와이프를 주동작에서 내리고 **크고 명확한 "다음/보기" 버튼**을 주동작으로.
- 입력 필드·터치타깃 확대(이미 button/input 사이즈 상향된 readability 작업과 정합).
- 출생지: 검색창보다 프리셋 칩 우선 노출.
- 진행표시(3/3)는 화면이 1개로 줄면 자연히 단순화.

## 4. 영향/리스크

- **정합성**: 밑단 리졸버·slug 공통이라 결과 계산 로직 변화 없음(입력 수집 UI만 재배치).
- **결정론 slug**: `toSlug`는 raw input 사용 — 관심주제를 흐름에서 빼도 slug/결과 불변(관심주제는 slug 비관여, 오늘운세 이름 fallback 메모 참고).
- **호환**: 기존 게스트 저장키·프로필 자동로드 경로는 유지(파괴적 변경 없음). 새 공용 키는 additive.
- **범위 경계**: `saju-intake-page.tsx`가 1713줄 단일 파일 — 스텝 병합 시 렌더 함수 분리(파일 정리) 동반 권장.
