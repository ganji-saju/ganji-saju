# 대운 풀이 LLM 생성 스펙

> ganjisaju.kr 대운 8섹션 풀이를 9개 대운 각각 고유하게 만들기 위한 완전한 프롬프트·스키마·후처리 규칙.
>
> **이 문서는 `naming-policy.md`의 어휘 정책을 따릅니다.** 충돌이 있으면 `naming-policy.md`가 우선.
>
> 짝이 되는 자료: `saju-total-review-llm-spec.md` (총평), `saju-score-spec.md` (점수), `saju-terms-dictionary.json` (용어 사전).

---

## 0. 이 문서가 풀려는 문제

지금 사이트의 9개 대운 풀이는 **천간/지지 글자와 12운성 라벨만 바뀌고 본문은 거의 동일한 템플릿**으로 보입니다. 특히 "관계" 섹션, "돈·커리어" 섹션, "실천 4단" 중 2~3개는 9개 대운에 같은 문장이 들어가 있어 사용자가 "내 사주가 아니라 일반론을 본다"고 느낍니다.

원인은 4계층 파이프라인(L1 입력 → L2 계산 → **L3 LLM 해석** → L4 UI) 중 **L3이 사실상 동작하지 않는 것**으로 보이는 데 있습니다. L2 계산값을 L4 템플릿에 직접 끼워넣고 부사어만 몇 개 바꾸는 식.

이 문서의 목적은 **L3을 제대로 동작시켜 9개 대운이 진짜로 다른 본문**을 갖게 하는 것입니다.

---

## 1. 핵심 설계 원칙

### 1-1. 대운의 고유성은 *3축 조합*에서 나온다

각 대운의 본문은 다음 세 축의 *조합 의미*가 드러나야 합니다.

| 축 | 값 | 의미 |
|---|---|---|
| **천간 오행** | 갑·을(목) / 병·정(화) / 무·기(토) / 경·신(금) / 임·계(수) | 들어오는 기운의 *성질* |
| **지지 12운성** | 장생/목욕/관대/건록/제왕/쇠/병/사/묘/절/태/양 | 그 기운의 *강도와 단계* |
| **일간 대비 십성** | 비견/겁재/식신/상관/편재/정재/편관/정관/편인/정인 | 본인 사주와의 *관계* |

여기에 **interactions**(원진·합·충·형 등 본인 사주와 대운 사이의 상호작용)를 추가하면 9개 대운이 자연스럽게 9개 다른 본문을 갖게 됩니다.

### 1-2. LLM에는 *이미 일상어로 번역된* 키워드만 전달

LLM에게 "묘신원진", "목욕지", "정인격" 같은 명리 용어를 주고 "쉽게 써줘"라고 하면 매번 결과가 다릅니다. LLM이 자기 식대로 풀어 쓰면서 한자도 노출하고요.

해결: 입력 JSON에 미리 일상어 풀이를 같이 넣고, **그 풀이만 써서 작성하라**고 지시합니다. 명리 용어 자체는 `_internal` 필드에 두고 본문에 쓰지 않도록.

### 1-3. 섹션별 호출 (병렬, 8회)

한 대운을 한 번에 다 생성하지 마세요. 8개 섹션을 각각 별도 호출하면:
- 각 섹션이 600~1000자 안에서 깊이 있게 나옴
- 한 섹션 실패해도 재생성 비용이 작음
- 병렬 호출로 응답 속도 빠름 (8섹션 → 1.5~3초)
- 섹션별로 다른 톤·길이 룰 적용 가능

### 핵심 포인트

- `_internal` 필드는 LLM이 본문에 쓰면 안 되는 데이터(한자, 명리 학술 용어). 시스템 프롬프트에 명시.
- 모든 핵심 키에 `_easy` 풀이가 함께 들어가 LLM이 자기 식대로 추측하지 않도록.
- `interactions`는 합·충·형·해·원진 등을 미리 계산해서 일상어로 변환한 것만 전달.
- `completes_yongsin` 같은 메타 정보는 본문에서는 "잘 쓰면 부족했던 부분을 채워준다"는 톤으로 쓸 수 있음.

## 3. 시스템 프롬프트 (모든 섹션 공통)

```text
당신은 친근하고 차분한 명리 해설가입니다. 사용자가 지나는 한 10년(대운)에 대해, 어렵지 않은 일상어로 풀어 씁니다.

[절대 규칙 — 한 번이라도 어기면 출력 폐기]
1. 한자(漢字)는 본문에 한 글자도 노출하지 않는다. _internal 필드의 값은 절대 본문에 쓰지 않는다.
2. 사주 전문 용어를 본문에 쓰지 않는다. 다음 단어 금지:
   - 천간, 지지, 일간, 일주, 월주, 시주, 연주, 시지, 월지, 연지
   - 격국, 정인격, 편관격, 식상격 등 모든 격국명
   - 용신, 신강, 신약, 강약, 순행, 역행, 대운, 세운, 월운
   - 장생지, 목욕지, 관대지, 건록지, 제왕지, 쇠지, 병지, 사지, 묘지, 절지, 태지, 양지
   - 비견, 겁재, 식신, 상관, 편재, 정재, 편관, 정관, 편인, 정인
   - 합, 충, 형, 파, 해, 원진, 공망, 신살, 양인, 도화, 역마, 화개, 교운기
   대신 입력 JSON의 _easy 필드에 있는 일상어 풀이를 그대로 또는 자연스럽게 변형해서 사용한다.
3. "결"이라는 단어 사용 엄격 제한 (`naming-policy.md` §9):
   - 섹션 제목·라벨·요약: 사용 금지
   - 본문: 섹션당 최대 1회, 정말 필요할 때만
   - 대신 "시기", "흐름", "기운", "리듬", "분위기", "사주", "성향", "방식"을 사용
4. 한 문장 길이는 60자 안팎. 80자를 넘기지 않는다.
5. 사용자 이름은 한 섹션에서 0~1회만 부른다. 부를 때는 성을 빼고 이름만 + "님".
6. "재호 님은…", "재호 님의…" 식의 반복 호명 금지.
7. 본문에 "대박", "비책", "암흑기", "텅장", "꿀팁" 같은 자극적·속어성 말 금지.
8. 막연한 위로("괜찮을 거예요", "잘 될 거예요") 금지. 반드시 구체적인 상황이나 행동을 짚을 것.

9. 오행 어휘 정책 (`naming-policy.md` §2):
   - 오행은 반드시 "X 기운" 형태로 표기:
     "목 기운 / 화 기운 / 토 기운 / 금 기운 / 수 기운"
   - 금지: "쇠의 결", "햇살의 결", "흙의 결", "물의 결", "새싹의 결"
     (자연 비유 + "결" 합성)
   - 금지: "결단과", "안정과", "열정과", "시작과", "지혜과" (구 X과 라벨)
   - 금지: "쇠", "햇살", "새싹" 등 자연 명사를 *단독으로* 오행 대신 사용
   - 첫 등장 시 짧은 의미를 옆에 붙일 수 있음:
     "금 기운(단단함과 결단)이 들어오면…" → 이후엔 그냥 "금 기운"

10. 십성 어휘 정책 (`naming-policy.md` §3):
    - 식신·정인·정관·편관 등 십성은 *원어 그대로* 사용
    - 첫 등장 시 짧은 설명: "식신(나누고 베푸는 별)"
    - 금지: "말의 기운", "생각 기운", "절제의 기운" 등 추상명사화

[구조 규칙]
- 각 섹션은 다음 구조: [한 줄 결론] → [왜 그런지 일상어 비유 또는 사용자 상황 매핑] → [구체적 액션 또는 관찰 포인트]
- 사용자 컨텍스트(context.relationship_status / occupation_status / concern)를 본문에 자연스럽게 녹여 쓸 것. 단, 어색하게 끌어다 붙이지 말 것.

[톤]
- 친근한 구어체 (~예요, ~죠, ~세요). 단정형(~다)은 피한다.
- 따뜻하지만 단호한 톤. 점쟁이 톤(과장·확정) 금지.
- 비유는 일상에서 가져온다 (가을, 강물, 씨앗, 무대, 운전 등 *흔한* 일상 사물).
  자연 명사 + "~처럼" 직유는 본문에 자연스럽게 녹는 한 OK ("햇살처럼 퍼지는",
  "강물처럼 깊어지는"). 단, 자연 명사 + "의 결" 합성은 금지 ("쇠의 결",
  "햇살의 결" 등). 추상어("기운의 흐름") 남용 금지.
- 오행 라벨은 항상 "X 기운" ("금 기운", "화 기운" 등)으로.

[출력 형식]
지정된 JSON 스키마를 정확히 따른다. 다른 텍스트는 출력하지 않는다.
```

## 4. 섹션별 사용자 메시지 (8개)

각 섹션마다 시스템 프롬프트 + 동일한 입력 JSON + 섹션 고유 지시문을 보냅니다. 출력은 해당 섹션의 JSON만.

### 4-1. 호명 (calling_line)

```text
입력 JSON을 참고해 사용자가 지금 어떤 10년을 지나고 있는지를 한 문장으로 알려주세요. 70자 이내.

- 현재 진행 중인 대운이면 "지금 진행 중인 ~ 의 10년은 …" 형태로 시작.
- 과거 대운이면 "~ 시절의 10년은 …" 형태로 시작.
- 미래 대운이면 "앞으로 다가올 ~ 의 10년은 …" 형태로 시작.
- 그 10년의 가장 핵심적인 한 가지 키워드를 일상어로 단정적으로 짚을 것.
- 예: "지금 진행 중인 22-31세의 10년은 밝게 드러나고 인정받는 시기예요."

출력은 다음 JSON만:
{ "calling_line": "..." }
```

### 4-2. 챕터 제목 (chapter_title)

```text
입력 JSON을 참고해 이 10년의 챕터 제목을 한 줄로 만들어주세요. 28자 이내.

- 3축 조합(천간 오행 + 12운성 + 십성)의 고유 의미를 담을 것.
- "대박 나는 비책", "암흑기 주의보" 같은 자극적 문구 금지.
- "운명을 내 편으로", "잘 풀리는 사람들의 공통 루틴" 같은 자기계발서 톤 금지.
- 차분하고 구체적인 묘사형 제목.
- 좋은 예: "회복과 정리가 끝나고 본격 도약이 시작되는 시기"
- 좋은 예: "조용히 자기 자리를 다지는 10년"
- 좋은 예: "말이 앞서고 마무리가 비기 쉬운 시기"

출력은 다음 JSON만:
{ "chapter_title": "..." }
```

### 4-3. 본문 (overview)

```text
입력 JSON을 참고해 이 10년의 전체 흐름을 2~3개 단락으로 풀어주세요. 총 400~550자.

[반드시 포함할 요소]
1. 들어오는 기운의 성질 (gan_easy + ji_easy)
2. 본인 사주와의 관계 (gan_sipsin_easy + ji_sipsin_easy 중 메인을 골라)
3. 사주를 보강하는지/과하게 만드는지 (completes_yongsin_meaning)
4. is_transition이 true면 교운기 의미를 한 문장으로 짚어줄 것

[형식]
- 첫 문장은 "지금 지나고 있는 22-31세의 10년은, 한마디로 [핵심 한 단어]이에요." 같은 단정적 한 줄 결론.
- 두 번째 문장부터 왜 그런지 풀이.
- 마지막 단락에 한 가지 조심할 부분 또는 챙길 부분.

출력은 다음 JSON만:
{ "overview": "..." }
```

## 6. 후처리 검증 (L3.5)

LLM 출력을 그대로 사용자에게 보여주기 전, 다음 검사를 통과해야 합니다.

```typescript
type ValidationResult = { ok: boolean; reasons: string[] };

function validateDaewoonOutput(text: string): ValidationResult {
  const reasons: string[] = [];

  // 1. 한자 누출 검사
  const hanjaMatches = text.match(/[\u4e00-\u9fff]/g);
  if (hanjaMatches && hanjaMatches.length > 0) {
    reasons.push(`한자 누출: ${[...new Set(hanjaMatches)].join(', ')}`);
  }

  // 2. 금지 용어 검사
  const bannedTerms = [
    '천간', '지지', '일간', '일주', '월주', '시주', '연주',
    '시지', '월지', '연지', '격국', '용신', '신강', '신약',
    '대운', '세운', '월운', '순행', '역행',
    '장생지', '목욕지', '관대지', '건록지', '제왕지', '쇠지',
    '병지', '사지', '묘지', '절지', '태지', '양지',
    '비견', '겁재', '식신', '상관', '편재', '정재',
    '편관', '정관', '편인', '정인',
    '합', '충', '형', '파', '해', '원진', '공망', '신살',
    '양인', '도화', '역마', '화개', '교운기'
  ];
  // "합"은 단독 검사 시 일상어("합치다")와 충돌하므로 "○합"/"합○" 패턴만
  const compoundBanned = ['천간합', '지지합', '삼합', '육합', '천간충', '지지충'];

  for (const term of bannedTerms) {
    if (text.includes(term)) {
      reasons.push(`금지 용어: ${term}`);
    }
  }

  // 3. "결" 빈도 검사 (naming-policy §9 적용 — 섹션당 최대 1회)
  const gyeolCount = (text.match(/결/g) || []).length;
  if (gyeolCount > 1) {
    reasons.push(`'결' 과다 사용: ${gyeolCount}회 (목표 1회 이하)`);
  }

  // 3-A. 어휘 정책 검증 (naming-policy.md §12)
  const forbiddenPatterns: Array<[RegExp, string]> = [
    [/(새싹|햇살|흙|쇠|물)의\s*결/g, '자연 비유 + "의 결"'],
    [/결단과|안정과|열정과|시작과|지혜과/g, '구 X과 라벨'],
    [/(표현|생각|절제|직관|돌봄|관찰|베푸는)의\s*기운/g, '십성 추상명사화'],
    [/(돌봄|표현|기준|단단함)의\s*결/g, '추상명사 + 결'],
    [/(표현|돌봄|재물|관계|기준)형\s*사주/g, 'X형 사주 신조어'],
  ];
  for (const [pattern, label] of forbiddenPatterns) {
    const matches = text.match(pattern);
    if (matches?.length) {
      reasons.push(`금지 어휘 (${label}): ${[...new Set(matches)].join(', ')}`);
    }
  }

  // 3-B. 오행 "X의 기운" 형태 검출 (X 기운으로 표기해야)
  const ohaengWrongPattern = /(목|화|토|금|수)의\s*기운/g;
  const ohaengWrongMatches = text.match(ohaengWrongPattern);
  if (ohaengWrongMatches?.length) {
    reasons.push(`오행 "X의 기운" 형태 금지 (X 기운으로 표기): ${[...new Set(ohaengWrongMatches)].join(', ')}`);
  }

  // 4. 자극적 문구 검사
  const clickbait = ['대박', '비책', '암흑기', '텅장', '꿀팁'];
  for (const word of clickbait) {
    if (text.includes(word)) {
      reasons.push(`자극적 문구: ${word}`);
    }
  }

  // 5. 문장 길이 검사 (평균 60자 안팎, 100자 초과 문장 없음)
  const sentences = text.split(/[.!?]\s+/).filter(s => s.length > 0);
  const tooLong = sentences.filter(s => s.length > 100);
  if (tooLong.length > 0) {
    reasons.push(`100자 초과 문장 ${tooLong.length}개`);
  }

  return { ok: reasons.length === 0, reasons };
}
```

검증 실패 시 처리:
- 1차 실패 → 같은 프롬프트로 1회 재생성
- 2차 실패 → 시스템 프롬프트의 위반 항목을 강조한 추가 지시문 붙여 재생성
- 3차 실패 → 운영자 알림 + fallback 텍스트 사용

운영 대시보드에서 매일 모니터링할 지표:
- 한자 누출률 (목표: 0.1% 이하)
- 금지 용어 위반율
- 평균 재생성 횟수
- 섹션별 평균 글자 수 분포

## 7. 9개 대운 동시 생성 시 호출 패턴

한 사용자에 대해 9개 대운(예: 갑오/을미/병신/정유/무술/기해/경자/신축/임인) × 8섹션 = 72회 LLM 호출이 됩니다. 비용·속도 관리가 중요해요.

### 권장 패턴

```
[1단계: 무료 사용자에게 즉시 보여줄 부분]
- 현재 대운(is_current=true)만 8섹션 전부 즉시 생성
- 나머지 8개 대운은 카드만 미리 보여주고 본문은 lazy load

[2단계: 사용자가 다른 대운 카드를 클릭/펼치면]
- 해당 대운의 8섹션을 병렬 호출
- 응답 도착 순서대로 화면에 스트리밍

[3단계: 캐싱]
- 결과를 (생년월일시 + 성별 + context 해시) 키로 캐시
- 같은 사용자가 다시 들어와도 재호출 없음
- 단, context(솔로/구직 중 등)가 바뀌면 재생성
```

### 비용 추정 (GPT-5 계열, 2026년 가격 근거 추정)

- 호출당 입력 토큰 ≈ 2,000 (시스템 프롬프트 + 입력 JSON + few-shot)
- 호출당 출력 토큰 ≈ 200~400 (섹션별)
- 8섹션 × 9대운 = 72호출
- 풀 생성 시 토큰 ≈ 200,000 input + 25,000 output

GPT-5-mini 또는 동급 모델로 충분하므로, 한 사용자당 풀 생성 비용은 약 $0.05~$0.15 추정. 캐싱하면 첫 1회만.

## 8. 통합 체크리스트

배포 전에 다음을 확인:

- [ ] 9개 대운 각각 LLM에 다른 입력 JSON이 들어가는지 (단순 글자 교체 아닌지)
- [ ] `_internal` 필드의 한자가 본문에 누출되지 않는지 (자동 검증)
- [ ] context (솔로/구직 중/사업·이직)가 본문에 자연스럽게 반영되는지
- [ ] 9개 대운의 "관계", "돈·커리어", "마무리" 섹션이 각각 다른 텍스트인지 diff 검사
- [ ] 실천 4단의 액션이 9개 대운에서 다른지 (1개 정도는 공통 허용, 3개는 고유)
- [ ] 챕터 제목 fallback("대박 나는 비책 …")이 한 번도 등장하지 않는지
- [ ] 운영 대시보드에 한자 누출률·금지 용어 위반율 지표 등록

## 9. 다음 단계 (이 문서 이후)

1. **세운(年運) 풀이도 동일 구조로**: 1년 단위 풀이는 대운보다 짧고 (200~300자), 12개월에 걸쳐 4분기 흐름을 짚는 형태로. 같은 3축 조합 원리.
