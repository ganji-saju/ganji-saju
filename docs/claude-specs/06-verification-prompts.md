## 검증 1: 회귀 테스트

**목적**: P0 작업·네이밍 마이그레이션·LLM 스펙 적용 과정에서 **건드리지 말아야 할 부분**이 그대로 유지됐는지 확인.

**왜 필요**: 대규모 텍스트 치환은 의도하지 않은 영역에 손이 가기 쉽습니다. 특히 정규식 일괄 치환에서 사주팔자 카드의 한자까지 같이 바뀌어버리는 사고가 자주 납니다.

### Claude Code 프롬프트

```
ganjisaju.kr 프로젝트에서 최근 작업(P0 버그 수정, 네이밍 마이그레이션, LLM 스펙 적용)
이후의 회귀(regression) 검증을 진행해줘.

## 확인할 영역 (반드시 변경되지 않아야 하는 부분)

### 1. 사주팔자 8글자 카드 — 한자 그대로 유지
- 총평 탭 사주팔자(四柱) 카드 영역에서 천간/지지 8글자가 한자로 표시되어야 함
- 예: 丙子 / 癸巳 / 己巳 / 丁卯
- 컴포넌트 파일과 렌더링 결과 모두 확인
- 만약 이 영역에 한자가 사라지고 한글만 나오면 P0 작업이 과도하게 적용된 것

### 2. 6분야 게이지 (총평/세운/상세 탭)
- 총운 / 직장·사업운 / 재물운 / 애정·연애운 / 인간관계운 / 컨디션·건강운
- 6개 게이지의 색상, 점수, 라벨, 게이지 바 표시가 변경 전과 동일한지
- 점수 산출 로직이 변경되지 않았는지

### 3. 결제·로그인·계정·정책 페이지
- 결제 페이지 안내 문구
- 로그인/회원가입 화면 텍스트
- 약관·개인정보처리방침
- 이 영역에 우연히 "결단과" 같은 단어가 있었다면 마이그레이션에 휩쓸렸을 수 있음

### 4. 명식 탭, 오행 탭, 성향 탭 (이번 작업 대상 외)
- 데이터 표시가 정상 동작하는지
- 강약 점수, 오행 비율, 신살 라벨 등 데이터 표시 영역의 깨짐 여부

### 5. UI 자산
- 시기 라벨 컬러 코딩 (성장기/전환기/표현기/기반기/결정기/준비기)
- "지금" 태그, "교운기 진입/마무리" 태그
- 월간 지도 캘린더 색상
- PEAK/PITFALL 라벨
- "안심하고 보기" 섹션 5개 카드 (저장됨, 다시 볼 수 있음, 다시 확인 가능, 시간·출생지 기준, 공포성 문구 없음)

## 작업 방식

1. 최근 작업의 git log/diff를 확인해 어떤 파일들이 수정됐는지 목록화
2. 위 1~5 영역과 관련된 파일이 수정 목록에 있는지 교차 확인
3. 의도되지 않은 변경이 발견되면 해당 파일·줄·변경 내용 보고

## 보고 형식

다음 표 형태로 정리해서 보고:

| 영역 | 상태 | 발견 사항 | 관련 파일 |
|------|-----|---------|----------|
| 사주팔자 카드 | PASS/FAIL | ... | ... |
| 6분야 게이지 | PASS/FAIL | ... | ... |
| 결제·로그인 | PASS/FAIL | ... | ... |
| 명식·오행·성향 탭 | PASS/FAIL | ... | ... |
| UI 자산 | PASS/FAIL | ... | ... |

FAIL이 하나라도 있으면 즉시 롤백 또는 수정 계획을 제시해줘.
PASS면 다음 검증 단계로 진행할 준비가 됐다고 알려줘.
```

**통과 조건**: 5개 영역 모두 PASS. FAIL이 1개라도 있으면 다음 단계로 진행하지 말고 우선 복구.

## 검증 2: P0 버그 6개 spot-check

**목적**: 진단서에 명시된 6개 P0 버그가 *각 위치에서* 실제로 수정됐는지, 그리고 *같은 패턴이 다른 자리에* 또 있지 않은지 확인.

**왜 필요**: Claude Code는 지시받은 자리만 고치는 경향이 있어 비슷한 패턴이 다른 곳에 또 있는 걸 놓치기 쉽습니다.

### Claude Code 프롬프트

```
ganjisaju.kr의 P0 버그 6개가 모두 수정됐는지, 그리고 동일 패턴이 다른 자리에
또 남아있지 않은지 확인해줘. 진단서(ganjisaju-comprehensive-diagnostic.md)의
§3-1 버그 6개가 기준이야.

## 버그별 점검 항목

### 버그 ①: "내 내 내 사주표" (변수 누수)
- 원래 발견 위치: 상세 7장 건강 리듬, 상세 1장 원국의 본질, 상세 9장 평생 활용 전략,
  세운 건강·생활 리듬 섹션
- 확인 1: 위 4개 자리에서 이제 "내 사주표" (또는 "이 사주") 한 번만 등장하는지
- 확인 2: 코드베이스 전체에서 정규식 `/내 내|내 내 내/` grep — 0건이 정답
- 확인 3: 템플릿 변수 정의에서 `{{user_pronoun}}` 또는 유사 변수가 한 문장에
  중복 삽입되는 패턴이 더 있는지

### 버그 ②: "근" → "기" 한자→한글 매핑 오류
- 원래 발견 위치: 상세 2장 한 줄 요약 ("근 내 핵심 기질"), 세운 1월 카드 태그 ("근표")
- 확인 1: 위 자리에서 이제 "기 내 핵심 기질", "기표"(또는 정확한 일주표/사주표 표기)로
  표시되는지
- 확인 2: 한자→한글 매핑 테이블이 있다면 10천간 전체 확인:
  甲=갑, 乙=을, 丙=병, 丁=정, 戊=무, 己=기, 庚=경, 辛=신, 壬=임, 癸=계
- 확인 3: 12지지 매핑도 동일하게 점검:
  子=자, 丑=축, 寅=인, 卯=묘, 辰=진, 巳=사, 午=오, 未=미, 申=신, 酉=유, 戌=술, 亥=해
- 확인 4: 사이트 본문에 "근 내", "근표" 정규식 grep — 0건이 정답

### 버그 ③: 영어 enum 누수 (weak/balanced/timing 등)
- 원래 발견 위치: 상세 2장 강약 점수 표시("결단과 3.1% · weak · 0.4점"),
  세운 이동·변화 섹션 ("명분과 timing을 같이 보는")
- 확인 1: 코드베이스 전체에서 본문에 노출되는 영어 단어 grep:
  weak, strong, balanced, timing, good, caution, pitfall, peak, early, mid, late
  (단, 변수명·주석·로그가 아닌 사용자 노출 문자열만)
- 확인 2: i18n 매핑 테이블이 있는지, 누락된 enum이 있는지

### 버그 ④: 세운 본문 종결문 접속 비문 ("챙기세요과", "핵심입니다를")
- 원래 발견 위치: 2026 흐름 메인 본문
- 확인 1: 정규식 `/[다요]과 |[다요]를 같이/` grep — 종결어미("~다"/"~요") 뒤에 바로
  접속 조사 "과"/"를"이 붙은 비문이 있는지
- 확인 2: 템플릿 패턴 `{판단A}과 {판단B}를 같이 조율` — 판단 자리에 들어가는 데이터가
  *명사구*인지 *종결문*인지 확인. 종결문이면 비문 발생

### 버그 ⑤: "커안쪽 결만" 5월 카드
- 원래 발견 위치: 세운 5월 카드 본문 ("바꾸고 싶은 마음이 커안쪽 결만 시험 운전이…")
- 확인 1: 5월 카드 본문이 자연스러운 한국어로 수정됐는지
- 확인 2: 12개 월 카드 본문 패턴이 비슷하게 변수 결합 오류를 일으킬 수 있는지
  (1~12월 모두 점검)

### 버그 ⑥: 챕터 제목 fallback ("대박 나는 비책, 이것만 지키면 흐름이 바뀝니다")
- 원래 발견 위치: 대운 을미·임인·계묘 3개 대운 카드
- 확인 1: 위 3개 대운의 제목이 이제 *서로 다른* 고유 제목인지
- 확인 2: 사이트 전체에서 "대박", "비책", "암흑기", "텅장" 키워드 grep —
  자극적 문구 잔존 여부 (있어도 됨 — 단, 의도된 카피인지만 확인)
- 확인 3: 9개 대운 카드 제목이 모두 *서로 다른지* (중복 0건)

## 보고 형식

각 버그별로 다음 형태:

### 버그 ①: 내 내 내 사주표
- 원래 위치 4곳 수정 여부: ✅/❌ (각 위치 파일·줄 명시)
- 정규식 grep 결과: 0건 / N건
- 동일 패턴 잔존 여부: 없음 / 있음 (위치 명시)
- 종합: PASS / FAIL

마지막에 전체 요약:
- 6개 버그 중 PASS: N개
- FAIL: N개 (각각 무엇인지)
- 추가 발견된 유사 패턴: N건 (목록)

FAIL 또는 추가 패턴이 있으면 즉시 수정 계획 제시.
```

**통과 조건**: 6개 모두 PASS + 추가 유사 패턴 0건.

## 검증 3: 네이밍 마이그레이션 일관성

**목적**: "결단과/안정과/열정과/시작과/지혜과" → 새 라벨(자연 비유 또는 다른 선택) 마이그레이션이 사이트 전체에서 **100% 일관**되게 적용됐는지 확인.

**왜 필요**: 일관성이 깨지면 사용자가 같은 개념인지 모릅니다. "한 곳은 쇠의 결, 다른 곳은 금 기운"이 되면 안 됨.

### Claude Code 프롬프트

```
ganjisaju.kr의 오행 네이밍 마이그레이션이 사이트 전체에서 100% 일관되게
적용됐는지 확인해줘. 이번 작업의 새 라벨 매핑이 무엇인지 먼저 알려주고,
그 매핑이 사이트 전체에 누락 없이 적용됐는지 검증해.

## 확인할 항목

### 1. 새 라벨 매핑 확인
- 다음 5개 오행에 대한 새 라벨이 정의된 파일을 찾아서 보여줘:
  - 木 (구 라벨: 시작과) → 새 라벨: ???
  - 火 (구 라벨: 열정과) → 새 라벨: ???
  - 土 (구 라벨: 안정과) → 새 라벨: ???
  - 金 (구 라벨: 결단과) → 새 라벨: ???
  - 水 (구 라벨: 지혜과) → 새 라벨: ???

### 2. 구 라벨 잔존 검사 (PASS 조건: 0건)
다음 정규식으로 사이트 본문·카드·태그·차트·UI 전체에서 검색:

```
결단과|안정과|열정과|시작과|지혜과
```

- 결과가 0건이면 PASS
- 1건이라도 있으면 정확한 파일·줄·맥락 보고

### 3. 합성 문구 점검
구 라벨이 합쳐진 말이 자연스럽게 마이그레이션됐는지:

- "결단과 · 안정과 보완 축이 살아나는…" 같은 합성 문구 검색
- 새 라벨로 바뀐 후 자연스러운 한국어가 됐는지 확인
- 예: "쇠의 결과 흙의 결 보완 축이…" → 자연스러움
- 안 좋은 예: "쇠의결안정과보완축이…" → 합성 사고

### 4. 숫자 표시 영역 점검
강약 점수, 오행 비율 같은 영역도 라벨이 일관되게 바뀌었는지:
- "결단과 3.1% · 약함 · 0.4점" → "쇠의 결 3.1% · 약함 · 0.4점" 식으로
- 5개 오행 모두 동일한 패턴으로

### 5. 새 라벨 자체의 일관성
사이트 전체에서 새 라벨이 *완전히 같은 형태*로 쓰이는지:
- 한 곳은 "쇠의 결", 다른 곳은 "쇠의결" (띄어쓰기 차이) — 불일치
- 한 곳은 "쇠의 결", 다른 곳은 "금의 결" — 불일치
- 한 곳은 "쇠의 결", 다른 곳은 "金 기운" (한자 잔존) — 불일치

### 6. 본문 자연스러움 spot-check
다음 위치에서 새 라벨이 자연스럽게 읽히는지 확인:
- 상세 1장 "원국의 본질" 본문
- 상세 2장 "기운의 균형" 본문 (구 라벨이 가장 많았던 곳)
- 상세 3장 "역할과 보완 힌트" 본문
- 세운 1장 "올해 흐름" 본문
- 대운 카드 9개의 본문

각 위치에서 새 라벨이 등장하는 문장을 하나씩 인용해서 자연스러움 평가.

## 보고 형식

```

- 木 → [새 라벨]
- 火 → [새 라벨]

- 金 → [새 라벨]
- 水 → [새 라벨]

- 정규식 grep 결과: N건

- 9장 본문의 문장 중 1~8장과 *동일하거나 80% 이상 유사한* 문장의 비율
  - 목표: 30% 이하 (이전: 80%+로 추정)
- 9장에서 새롭게 도출된 *평생 의사결정 원칙*이 3~5개 명시적으로
  나열되는가
  - 예: "강한 흐름이 들어올 때마다 ~를 우선" 같은 일반화된 원칙
- 9장이 사주 *전체*를 관통하는 메시지를 주는가, 아니면 단순 요약인가

### 4. 챕터 도입 문장의 다양성

이전엔 거의 모든 챕터가 "차분히 흐름을 읽어보면…"으로 시작했음.

- 9개 모두 *서로 다른 패턴*으로 시작하는가
- "차분히 흐름을 읽어보면…" fallback 등장 횟수 (목표: 0)
- 각 챕터가 자기 주제에 맞는 *고유한* 도입을 갖는가

### 5. 4장 vs 5장 도입 문장 비교

이전엔 4장(관계)·5장(재물) 도입 문장이 100% 동일했음.
새 생성에서 이 둘이 완전히 다른 도입을 갖는지 확인하고,
4장 첫 문단과 5장 첫 문단을 나란히 인용해서 보여줘.

- 구체적 액션 1개 이상

- 4-B 다양성 FAIL: 시스템 프롬프트에 "다른 챕터와 동일 문장 금지" 규칙 추가 + few-shot 보강
- 4-C 톤 FAIL: 시스템 프롬프트의 금지 규칙 강화 + 일반인 테스트로 약점 패턴 발견

### 검증 5 (일반화) FAIL 시
약점 사주 유형 1~2개를 골라 few-shot 예시 추가 → 재검증

### 검증 6 (운영) FAIL 시
- 응답 속도 문제: 병렬 호출 확인, 모델 다운그레이드(mini 모델) 검토
- 비용 초과: 토큰 절감, 캐시 적극 활용
- 재생성 비율 높음: 시스템 프롬프트 규칙 강화
